# Add a Strata migration

The test schema is built from the **models** (`create_all`); the real DB
(`construction_db`) from **migrations**. They must match — a model-only or
migration-only change is a bug we catch in review. Do BOTH.

## Steps

1. **Model first** — edit/add the mapped class under `backend/app/models/` and
   export it in `app/models/__init__.py`. Mirror exactly what the migration will
   create (columns, nullability, indexes, FKs). Don't add a column-level
   `unique=True` AND a separate unique `Index` for the same column — pick one
   (we use the explicit `Index(..., unique=True)` to match the migration).

2. **Find the current head**:
   `UV_LINK_MODE=copy uv run alembic heads`

3. **Write the migration** in `backend/alembic/versions/` by hand, matching the
   existing files' style (module docstring explaining the change, `revision`,
   `down_revision` = current head, `upgrade()` + `downgrade()`). These are
   hand-authored, not autogenerated. For Postgres enums referenced in
   `bulk_insert`, use the real enum type with `create_type=False`.

4. **Apply to construction_db**:
   `UV_LINK_MODE=copy uv run alembic upgrade head`

5. **Verify** the suite still passes (it builds the schema from the model):
   `UV_LINK_MODE=copy uv run pytest -q` (in the background).

## Checklist before calling it done

- [ ] Model + migration describe the same DDL
- [ ] Model exported in `app/models/__init__.py`
- [ ] `down_revision` points at the previous head
- [ ] `alembic upgrade head` succeeded against `construction_db`
- [ ] Full suite green
